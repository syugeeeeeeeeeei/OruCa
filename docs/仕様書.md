## OruCa 技術仕様書（2026-03 改訂）

### 1. 概要

OruCa は FeliCa カードを使って研究室メンバーの在室状況を管理するシステム。
構成は `web`(Nginx) / `api`(Node.js + WebSocket) / `mysql` / `nfc` の4サービス。

今回の改訂で、以下を実施した。

- MySQL ストアドプロシージャ依存を廃止し、主要ロジックを API 層へ移管
- 管理操作を WebSocket 接続単位の認証付きに変更
- Slack設定を WebUI から編集可能にし、DBへ暗号化保存
- バックアップ作成/一覧/リストアを WebUI から実行可能に変更

---

### 2. システム構成

- `web`
  - React ビルド成果物の配信
  - `/socket` を `api:3000` に WebSocket プロキシ
  - `/api/*` を `api:3000` に HTTP プロキシ
- `api`
  - WebSocket メッセージ処理（在室更新、管理認証、管理操作）
  - DBアクセス、Slack通知、バックアップ操作
- `mysql`
  - `users` / `logs` / `app_settings` を永続化
- `nfc`
  - FeliCa を読み取り、`log/write` を API に送信

---

### 3. DB 設計

#### 3.1 テーブル

- `users`
  - `student_ID` (PK)
  - `student_Name`
  - `student_token`（認証トークン）
- `logs`
  - `student_ID` (PK, FK -> users.student_ID)
  - `isInRoom`
  - `updated_at`
- `app_settings`
  - `id` (PK, 固定値 1)
  - `slack_channel_id`
  - `slack_bot_token_encrypted`
  - `slack_bot_token_iv`
  - `slack_bot_token_auth_tag`
  - `updated_at`

#### 3.2 ビュー

- `student_token_view`
- `student_log_view`
- `student_name_view`

#### 3.3 廃止事項

- `insert_or_update_log`
- `update_student_name`
- `get_student_token`

上記ストアドプロシージャは廃止し、同等処理を API の TypeScript 実装へ移行した。

補足: 既存DB向けに API 起動時に `app_settings` テーブルの存在確認と初期行投入を実施する。

---

### 4. 認証仕様

#### 4.1 基本仕様

- 管理ログインは「ユーザー名=学籍番号」「パスワード=固定パスワード」
- ただしトークン生成は `固定パスワード + 学籍番号` を用いる
- これにより運用上の入力は固定PWのまま、内部的には学籍番号依存の検証になる

#### 4.2 トークン生成

- Salt: `SHA256(student_ID)`
- DerivedPassword: `ADMIN_FIXED_PASSWORD + student_ID`
- Token: `SHA256(student_ID + DerivedPassword + Salt)`

#### 4.3 セッション管理

- WebSocket 接続ごとに「管理者認証済み」フラグを保持
- 未認証で管理操作を呼ぶと `type: "unauthorized"` を返す

#### 4.4 互換性

- 旧方式トークン（固定パスワードのみで生成）も認証時にフォールバック対応
- ユーザーが次回タップした時点で新方式トークンへ更新される

### 5. WebSocket API

#### 5.1 既存メッセージ

- `log/fetch`
- `log/write`
- `user/auth`
- `user/update_name`（認証必須）
- `user/delete`（認証必須）
- `user/fetchToken`（認証必須）

#### 5.2 追加メッセージ

- `admin/slack/get`
- `admin/slack/update`
- `admin/backup/create`
- `admin/backup/list`
- `admin/backup/restore`
- `unauthorized`（未認証アクセス時）

#### 5.3 `log/write` 処理

API 側トランザクションで実施する。

1. `users` に初回登録（未存在時）
2. `logs.isInRoom` をトグル（初回は `TRUE`）
3. 在室データをブロードキャスト
4. Slack設定が有効なら通知投稿

---

### 6. Slack 設定

- 管理画面からチャンネルID・Botトークンを設定
- Botトークンは `AES-256-GCM` で暗号化して `app_settings` に保存
- 復号鍵は `SLACK_TOKEN_ENC_KEY`（環境変数）
- 画面にはトークン平文を再表示しない

---

### 7. バックアップ仕様

- API が `mysqldump` / `mysql` を実行して操作
- 保存先は `BACKUP_DIR`（既定 `/backups`）
- WebUI から以下を実行可能
  - バックアップ作成
  - バックアップ一覧取得
  - バックアップリストア

#### 7.1 リストア安全策

- `confirmRestore=true` の明示確認が必要
- 実行前に自動で `pre-restore-*.sql` を作成
- リストアは同時実行不可（ロック）

---

### 8. 管理画面

既存デザイン（配色・フォント・ヘッダ構造）を維持し、`/admin/settings` に以下を追加。

- Slack通知設定パネル
- MySQLバックアップ管理パネル

既存の「氏名編集 / ユーザー削除」UI は維持。

---

### 9. 環境変数

- `MYSQL_DATABASE`
- `MYSQL_USER`
- `MYSQL_PASSWORD`
- `MYSQL_ROOT_PASSWORD`
- `ADMIN_FIXED_PASSWORD`（必須）
- `SLACK_TOKEN_ENC_KEY`（Slackトークン保存時に必須）
- `BACKUP_DIR`（既定 `/backups`）
- `APPDATA_ROOT`

---

### 10. 主要フロー

#### 10.1 カードタップ

1. `nfc` が `log/write` を送信
2. `api` が DB 更新（初回登録 + 在室トグル）
3. `api` が全クライアントへ `log/fetch` ブロードキャスト
4. Slack設定が有効なら通知投稿

#### 10.2 管理ログイン

1. 管理画面から `user/auth` 送信
2. `api` が `student_token` と入力PWを照合
3. 成功時に当該 WebSocket 接続を認証済み化

#### 10.3 バックアップリストア

1. 管理画面で対象バックアップを選択し確認
2. `admin/backup/restore` 送信
3. API が自動退避バックアップ作成
4. API が対象SQLをリストア
5. API が最新在室データをブロードキャスト
